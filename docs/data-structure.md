# ECG Data Structure

This document describes the layout and file format of the pathology
dataset shipped with CardioSimulator. It supersedes the legacy
`Parts/` + `Series/` layout (see [Appendix A](#appendix-a-legacy-format)).

The dataset is generated by `Data/build_pathologies.py`, packaged as
`Data/Pathologies.zip`, and consumed by the Android app's ECG playback
pipeline.

---

## 1. Overview

One file per pathology, all 12 standard ECG leads inside, raw ADC
samples as a continuous integer stream — no part boundaries, no
cross-file references.

```
Data/
├── Data/
│   └── Pathologies/             ← generated output (also bundled in zip)
│       ├── manifest.txt         ← dataset metadata + index
│       ├── 1abblock.dat
│       ├── 2abblock1.dat
│       ├── ...
│       └── wpwrightlat.dat
├── Pathologies.zip              ← shippable bundle (manifest + 56 .dat files)
└── build_pathologies.py         ← regenerator
```

### Headline numbers (version 1.0)

| Metric                | Value   |
|-----------------------|---------|
| Pathologies           | 56      |
| Total lead streams    | 666     |
| Total samples         | 586,532 |
| ADC baseline          | 1024    |
| Leads per pathology   | 12 (one exception: `emd` has 6) |
| Encoding              | UTF-8   |
| Line endings          | LF      |

## 2. Pathology file format (`<pathology>.dat`)

Plain UTF-8 text. One `key:value` entry per line. Blank lines separate
the header block from each lead block, and lead blocks from one another.

### 2.1 Grammar

```
<file>          ::= <header> <blank> ( <lead-block> <blank> )+
<header>        ::= "pathology:" <id>      NL
                    "title:"     <en-text> NL
                    "name:"      <ru-text> NL
                    "leads:"     <int>     NL
<lead-block>    ::= "lead:"   <lead-name>       NL
                    "count:"  <int>             NL
                    "points:" <int>(,<int>)*    NL
<lead-name>     ::= "I" | "II" | "III" | "aVR" | "aVL" | "aVF"
                  | "V1" | "V2" | "V3" | "V4" | "V5" | "V6"
<blank>         ::= NL
NL              ::= "\n"
```

### 2.2 Field semantics

**Header**

| Key         | Type   | Meaning                                                                    |
|-------------|--------|----------------------------------------------------------------------------|
| `pathology` | string | Stable pathology slug; also the file's basename. ASCII alphanumeric + `_`. |
| `title`     | string | English display name, without lead suffix.                                 |
| `name`      | string | Russian display name (UTF-8), without lead suffix.                         |
| `leads`     | int    | Number of lead blocks present in this file (always 12, except `emd` = 6).  |

**Lead block**

| Key      | Type   | Meaning                                                                              |
|----------|--------|--------------------------------------------------------------------------------------|
| `lead`   | string | Lead identifier from the fixed 12-lead vocabulary above.                             |
| `count`  | int    | Number of integers on the following `points:` line; equals `len(points.split(','))`. |
| `points` | ints   | Comma-separated raw ADC samples, no whitespace, baseline-centered on `1024`.         |

### 2.3 Lead ordering

Lead blocks appear in the canonical order:

```
I, II, III, aVR, aVL, aVF, V1, V2, V3, V4, V5, V6
```

If a lead is absent (only `emd` today, which lacks the precordial
V1–V6), its block is simply omitted — the file does not contain a
placeholder.

### 2.4 Sample values

- Type: signed 32-bit integer (in practice all values fit in `int16`).
- Baseline (isoelectric line): `1024`.
- Range observed: roughly `[896, 1649]` across the corpus.
- Sampling: the source data is unit-stepped; consumers treat the array
  as a uniform time series at the application's chosen rate (the rate
  is **not** encoded in the file — it is fixed by the playback engine).

### 2.5 Example (truncated)

```
pathology:tachpm
title:Atrial tachycardia, and pacemaker migration
name:Предсердная тахикардия и миграция водителя ритма
leads:12

lead:I
count:2262
points:1024,1024,1024,1023,1023,1022,1021,1020,1019,1018,...,1024

lead:II
count:2782
points:1024,1024,1024,1024,1024,1024,1024,...,1024

... (10 more lead blocks)
```

## 3. `manifest.txt`

Dataset-level metadata + per-pathology index. Same `key:value` syntax
as the data files. Always the first file you should read — it lets a
consumer validate the bundle and build a table of contents without
scanning every `.dat`.

### 3.1 Header keys

| Key                     | Example                              | Notes                                              |
|-------------------------|--------------------------------------|----------------------------------------------------|
| `version`               | `1.0`                                | Bumped on any format change. Consumers should check. |
| `created`               | `2026-05-21T07:47:48`                | ISO-8601 local time of generation.                  |
| `format`                | `lead-grouped raw ADC samples`       | Human label for the layout.                          |
| `encoding`              | `utf-8`                              | Applies to all files in the bundle.                  |
| `line_endings`          | `lf`                                 |                                                     |
| `baseline`              | `1024`                               | Isoelectric ADC value.                              |
| `lead_order`            | `I,II,III,aVR,aVL,aVF,V1,V2,V3,V4,V5,V6` | Canonical order.                                |
| `pathologies`           | `56`                                 | Number of `.dat` files in the bundle.               |
| `total_lead_streams`    | `666`                                | Σ leads across all pathologies.                     |
| `total_samples`         | `586532`                             | Σ ADC samples across the entire dataset.            |
| `source`                | `built from legacy Parts/ + Series/` | Provenance.                                         |
| `skipped_missing_parts` | `4`                                  | Legacy refs that did not resolve (see §5).          |
| `skipped_unbaked_parts` | `5`                                  | Legacy parts with NAN/empty samples (see §5).       |

### 3.2 Per-pathology summary

Below the header, one line per pathology (sorted alphabetically):

```
pathology:<id>;leads:<n>;samples:<total_samples>;title:<en>
```

Example:

```
pathology:tachpm;leads:12;samples:31568;title:Atrial tachycardia, and pacemaker migration
pathology:emd;leads:6;samples:2412;title:Electromechanical dissociation (EMD)
```

## 4. Zip bundle (`Pathologies.zip`)

- Layout: flat — all files at the zip root, no nested directories.
- Compression: `DEFLATE`.
- Always contains `manifest.txt` plus exactly `<pathologies>` `.dat` files.
- Current size: ~445 KB.

Recommended consumer flow:

1. Open zip, read `manifest.txt` first.
2. Check `version` is supported.
3. Build pathology → file index from the per-pathology lines.
4. Lazily open individual `.dat` entries on demand.

## 5. Known data quirks (carried over from legacy)

These are upstream data-quality issues in the source `Parts/` + `Series/`
folders, not bugs in the generator:

- **Unbaked parts (5)** — legacy parts with `amplitude:NAN`,
  `duration:Long.MIN_VALUE`, empty `points:`. Includes `SIN_I`,
  `AT_TH_WP_I_4`, and 3 others. The generator silently skips them; the
  surrounding samples are still concatenated, so the lead stream is
  slightly shorter than the sum of nominal part durations would suggest.
- **Missing parts (4)** — series files reference part identies that
  contain Windows-illegal filename characters (`/`, `:`) and therefore
  have no corresponding file on disk:
  `AF_75/4:1_I`, `AF_75/4:1_II`, `ATRFLU_75/4:1_V2`, `ATRFLU_75/4:1_V6`.
  Affects pathologies `fib` and `atrflu`. Skipped silently.
- **Title artifacts** — a few titles contain `?` placeholders where the
  legacy CP1251 source had untranslatable characters (e.g.,
  `2° AV Block Mobitz ? aVF CD3`). These are present in the source
  Series files and pass through unchanged.

## 6. Rebuilding the dataset

The dataset is regenerated by the Python script:

```
C:\VLN_Project\Data\build_pathologies.py
```

Requirements: Python 3.9+ (no third-party deps).

```powershell
cd C:\VLN_Project\Data
python build_pathologies.py
```

The script:

1. Walks `Data/Series/`, groups files by `pathology`.
2. For each `(pathology, lead)` pair, reads the ordered list of part
   identies from line 1 of the series file.
3. Concatenates the raw `points:` array of each referenced
   `Data/Parts/<identy>` file in order.
4. Writes `Data/Pathologies/<pathology>.dat` per §2.
5. Writes `Data/Pathologies/manifest.txt` per §3.
6. Zips the folder into `Data/Pathologies.zip` per §4.

The script is idempotent and safe to rerun.

## Appendix A: Legacy format

The original on-disk layout, retained at `Data/Data/Parts/` and
`Data/Data/Series/` for reference, was:

- **`Parts/<identy>`** — one file per "anchor segment", encoding both
  editable curve handles (line 1 `source:`) and the baked ADC sample
  array (line 6 `points:`).
- **`Series/<russian name with lead>`** — one file per
  `(pathology, lead)`, holding only references to parts plus placement
  metadata.

Legacy files used Windows-1251 encoding for Russian text, CRLF line
endings, and a comma as decimal separator in floating-point fields
(e.g., `amplitude:0,87...`). These quirks are eliminated in the new
format: everything is UTF-8 / LF / dot-decimal (where applicable —
the data values themselves are integers).

The legacy folders may be removed once all consumers (Android app,
desktop editor) are migrated to read `Pathologies.zip`.
